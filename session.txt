  核心流程

  输入：16位字母/数字水印文本、冗余份数 repl、原图、可选蒙版。

  嵌入规则
  - 水印(16字节) + ReedSolomon ECC(16字节) → 转为 bit 序列
  - 按 BLOCK_SIZE（8/12/16）在 alpha>0 区域划分 DCT 块网格（左→右，上→下）
  - 约束：repl × 单份(水印+ECC)bit长度 ≤ 总 DCT 块数
  - 一块嵌一 bit，使用策略1

  解码规则
  - 先对单份数据做 ECC 校验解码
  - 再对所有冗余份做投票后 ECC 解码
  - 取解码结果中出现最多者作为最终水印

  策略1（中频系数对调制）

  选中频一对位置 A/B，按 delta：
  - bit==1：调整使 coef[A] + delta ≤ coef[B]（A→B 增大差）
  - bit==0：调整使 coef[A] - delta ≥ coef[B]（A→B 减小差）

  拍照矫正管线

  去畸变 → SIFT/ORB 匹配 → findHomography(RANSAC) → warpPerspective 粗配准 →
  Phase Correlation 估计残余位移 → warpAffine 平移补偿 → ECC 精配准 → 亚像素对齐
  → 解码。

⏺ 已读完更新后的 CLAUDE.md。主要变化是新增并细化了 解码规则
  一节。我的理解更新如下：

  解码规则（更新重点）

  每份独立解码（优先路径）
  读Block → 判bit → 按repl分组 → 每份单独ECC
     ├─ 成功 → 记录到"成功结果集"
     └─ 失败 → 保留（暂不用）

  汇总
  所有成功结果 → 统计频率 → 取最多者 → 输出

  兜底（仅当所有份都 ECC 失败时）
  Bit Vote（逐bit多数投票） → 再做一次 ECC

  bit 判定不使用 delta（delta 只在嵌入时用于拉开能量差）：
  coef[B] > coef[A] → 1
  否则           → 0

  关键点确认

  1. 解码判 bit 无阈值：只比较 coef[B] 与 coef[A]
  的大小（硬判决），与嵌入方向一致。
  2. 两级容错：先"逐份 ECC + 频率投票"，全失败才降级到"逐 bit 投票 +
  ECC"——避免个别份的比特错误污染多数结果。
  3. 策略 2（第65行）仍为空，暂不实现。

  在已知模板的前提下，不需要"先抠图再矫正"。一个单应矩阵 H（photo→template）配合
  warpPerspective(photo, H, 模板尺寸)，一次就同时完成两件事：

  - 矫正：把目标区域拉正到模板坐标系；
  - 抠图：目标以外的背景在映射后落到画布之外，被自动丢弃。

  你现在的 align.pipeline 其实已经在做这件事——detect_and_match(photo, template)
  只会在目标上找到与模板匹配的特征点，背景点与模板对不上，被 RANSAC
  当外点剔除；最后 warp 到模板尺寸，背景自然裁掉。所以"有背景"本身不是新问题，问
  题是匹配足够稳时它就成立，不稳时才失败。

  什么时候会失败 → 需要增强

  目标在画面里太小 / 太斜 / 模糊 / 反光 / 低纹理时，匹配点不足，H
  估计错误或退化。增强手段：

  1. 单应质量校验（关键补强）：检查内点数、四角是否凸四边形、面积占比、长宽比是
  否接近模板。不合格就判"未检出目标 → 提示重拍"，而不是硬解出乱码。
  2. 两级定位：先用粗匹配得目标包围盒 → 裁剪出带余量的 ROI → 在 ROI
  内精匹配。减少背景干扰、提升精度。
  3. 强化特征：加大 nfeatures、多尺度、调 Lowe ratio。

  三条可选路线（各有取舍）

  ┌──────────────┬────────────────────┬─────────────────┬──────────────────┐
  │     路线     │        做法        │      优点       │       代价       │
  ├──────────────┼────────────────────┼─────────────────┼──────────────────┤
  │ A 纯内容特征 │ 强化 pipeline +    │ 不改被打印素材  │ 内容纹理不足/反  │
  │ （复用现有） │ 单应校验 +         │ ，投入小        │ 光时仍可能失败   │
  │              │ 两级定位           │                 │                  │
  ├──────────────┼────────────────────┼─────────────────┼──────────────────┤
  │ B            │                    │                 │ 要求目标有清晰矩 │
  │ 轮廓四边形（ │ Canny→找最大四边形 │ 不依赖模板      │ 形边界；90°/180° │
  │ 文档扫描式） │ →透视到已知长宽比  │                 │  朝向有歧义，需  │
  │              │                    │                 │ 再用模板定向     │
  ├──────────────┼────────────────────┼─────────────────┼──────────────────┤
  │              │ 在被打印图四角加   │ 几乎不受内容和  │                  │
  │ C 基准标记（ │ ArUco / 定位角 /   │ 背景影响，工业  │ 要改动要打印/展  │

⏺ 这行是把"模板的四个角"通过单应矩阵投影到拍照图上，从而框出目标在照片里的位置。
  逐部分拆解：

  上下文回顾

  前面几步：
  pts_small, pts_tpl = detect_and_match(small, template)   # 匹配点对
  H_tpl2small, mask = find_homography(pts_tpl, pts_small)   # 模板 -> 缩小拍照图
  的单应
  H_tpl2small 是个 3×3
  单应矩阵，作用是把模板坐标系里的点，映射到缩小后的拍照图坐标系。

  这行做的事

  1. _template_corners(template.shape)
  返回模板的四个角点，顺序 [左上, 右上, 右下, 左下]：
  [[0,0], [w,0], [w,h], [0,h]]   # 形状 (4,1,2)，w/h 是模板宽高

  2. cv2.perspectiveTransform(pts, H)
  对每个点做透视变换：把 (x,y) 变成齐次坐标 [x,y,1]，乘以

  [x', y', s] = H · [x, y, 1]ᵀ

  于是模板的四角被投影到"缩小拍照图"里——也就是目标那张图在照片中的四个角的位置。

  3. .reshape(-1, 2)
  把 OpenCV 要求的 (4,1,2) 形状压平成 (4,2)，方便后续用。

❯ np.linalg.norm(tr - tl) 是什么意思?                                           
  ⎿  ⧉ Selected 1 lines from photo_watermark/align/locate.py in Visual Studio 
     Code

⏺ np.linalg.norm(tr - tl)
  是求两个角点之间的直线距离（欧氏距离）——这里就是四边形上边的长度（像素）。

  拆解

  tr 和 tl 是两个角点坐标，各是一个 [x, y]：
  tl = [x_tl, y_tl]   # 左上 top-left
  tr = [x_tr, y_tr]   # 右上 top-right

  1. tr - tl —— numpy 逐元素相减，得到从 tl 指向 tr 的向量：
  tr - tl = [x_tr - x_tl,  y_tr - y_tl]   # = [dx, dy]

  2. np.linalg.norm([dx, dy]) —— 求这个向量的模长（L2 范数），即勾股定理：
  √(dx² + dy²)
  也就是 tl 到 tr 两点间的直线像素距离 = 上边的长度。

  在 _quad_aspect 里干嘛用

  tl, tr, br, bl = quad
  width  = 0.5 * (norm(tr - tl) + norm(br - bl))   # 上边、下边长度的平均 = 宽
  height = 0.5 * (norm(bl - tl) + norm(br - tr))   # 左边、右边长度的平均 = 高
  return width / height                             # 长宽比

  因为透视后四边形的对边不完全等长，所以上下两条边取平均当宽、左右两条边取平均当
  高，再算宽/高比值。这个比值拿去和模板的长宽比对比（validate_quad
  里），偏差太大就判定"这不是我们的目标"。

  一句话:norm(A - B) = A、B 两点的像素距离，这里用来量四边形的边长。

❯ 这里为什么要区别使用  cv2.warpPerspective, cv2.warpAffine 两种方法?           
  ⎿  ⧉ Selected 1 lines from photo_watermark/align/ecc.py in Visual Studio Code

⏺ 因为这两个函数吃的变换矩阵不一样，而不同阶段用的变换类型（自由度）也不一样。选
  哪个由"你要做的变换有几个自由度、矩阵是几行几列"决定。

  两者的区别

  ┌────────┬──────────────────────────┬──────────────────────────┐
  │        │      cv2.warpAffine      │   cv2.warpPerspective    │
  ├────────┼──────────────────────────┼──────────────────────────┤
  │ 矩阵   │ 2×3 仿射矩阵 M           │ 3×3 单应矩阵 H           │
  ├────────┼──────────────────────────┼──────────────────────────┤
  │ 自由度 │ 6（平移+旋转+缩放+错切） │ 8（再加透视）            │
  ├────────┼──────────────────────────┼──────────────────────────┤
  │ 能表达 │ 平行线保持平行           │ 平行线可汇聚（近大远小） │
  ├────────┼──────────────────────────┼──────────────────────────┤
  │ 计算   │ 无除法，快               │ 每点要除以第三分量，稍慢 │
  └────────┴──────────────────────────┴──────────────────────────┘

  关键：仿射变换保持平行，处理不了拍照那种"矩形拍成梯形"的透视；只有 3×3
  单应才能矫正透视，得用 warpPerspective。

  在这个项目里各自用在哪

  ① 透视矫正 → 必须 warpPerspective（homography.py / pipeline.py）
  拍照有透视畸变，findHomography 得到的是 3×3 单应 H，只能用 warpPerspective
  把梯形拉回矩形：
  warped = cv2.warpPerspective(roi, H, (w, h))   # H 是 3×3

  ② 纯平移补偿 → 用 warpAffine（phase.py 的 translate）
  相位相关只估出一个整体平移 (dx, dy)，这是最简单的变换，构造一个 2×3
  平移矩阵就够，没必要动用透视：
  M = np.float32([[1, 0, -dx],
                  [0, 1, -dy]])                   # 2×3
  cv2.warpAffine(image, M, (w, h))

  ③ ECC 精配准 → 按 warp_mode 二选一（ecc.py）
  findTransformECC 支持四种运动模型，矩阵形状不同，所以代码里分支：
  if warp_mode == cv2.MOTION_HOMOGRAPHY:      # 3×3
      warp = np.eye(3, 3, ...)
      cv2.warpPerspective(..., warp, ..., WARP_INVERSE_MAP)
  else:                                        # TRANSLATION/EUCLIDEAN/AFFINE 
  都是 2×3
      warp = np.eye(2, 3, ...)
      cv2.warpAffine(..., warp, ..., WARP_INVERSE_MAP)
  - MOTION_HOMOGRAPHY 是 8 自由度、3×3 → warpPerspective；
  - MOTION_TRANSLATION / EUCLIDEAN / AFFINE 都是 ≤6 自由度、2×3 → warpAffine。

  一句话总结

  矩阵是 3×3（含透视）用 warpPerspective，矩阵是 2×3（仿射/平移/欧氏）用 
  warpAffine。粗矫正要去透视 → 前者；相位相关只补平移 → 后者；ECC
  则看当前用的运动模型是哪种，矩阵几维就配哪个函数。

  补充一点：默认 refine_ecc 用的是 MOTION_TRANSLATION（走 warpAffine），因为前面
  warpPerspective 已经把透视矫正好了，ECC
  这步只需消除残余的亚像素平移，用最少自由度的模型最稳、最不容易发散。
